# 総合計画: プレイヤーの移動と攻撃制御機能の拡張

## 1. 目的

プレイヤーキャラクターの「攻撃中移動不可」と「移動中に攻撃した場合は停止」の機能を実装し、ゲームプレイの整合性を向上させる。

## 2. ToDoリストと推奨される着手順序

以下のToDoリストに基づいて作業を進めます。完了済みのToDoは次のステップに進むための前提条件となります。

1.  **ToDo 1: コード規約の理解** (AIによって完了済み)
    *   `Assets/Scripts/CodeGuidelines.md` を読み、プロジェクトのコーディング標準を把握。
    *   **考慮事項**: 以降のすべてのコード修正はこの規約に準拠する必要がある。

2.  **ToDo 2: 既存スクリプトの特定** (AIによって完了済み)
    *   キャラクターの移動 (`PlayerMover.cs`) と攻撃 (`PlayerAttacker.cs`) を制御する主要なスクリプトを特定。
    *   **考慮事項**: 特定されたスクリプトが本当に正しいかどうか、必要に応じて他の関連スクリプトも調査する。

3.  **ToDo 3: 移動・攻撃ロジックの理解** (AIによって完了済み)
    *   特定されたスクリプトの現在の実装を詳細に分析し、移動と攻撃のメカニズムを把握。
    *   **考慮事項**: どこにフックポイントがあるか、変更が他のシステムに与える影響などを事前に評価する。

4.  **ToDo 4: 攻撃中移動不可機能の実装計画** (AIによって完了済み)
    *   「攻撃中移動不可」と「移動中に攻撃した場合は停止」の機能を実装するための具体的な手順とコード変更箇所を計画。
    *   **考慮事項**: 循環参照の回避、責務の分離、アニメーションとの連携（攻撃終了のタイミング）などを考慮した設計を行う。

5.  **ToDo 5: コード修正の実装** (ユーザーによる手動実装、現在保留中)
    *   ToDo 4で策定した計画に基づき、`PlayerAttacker.cs`、`PlayerMover.cs`、および `PlayerManager.cs` のコードを修正。
    *   **考慮事項**:
        *   コード規約 (`Assets/Scripts/CodeGuidelines.md`) に厳密に準拠すること。
        *   変更は小さく、段階的に適用し、各変更後にUnityエディタで簡単な確認を行うことで、問題の早期発見に努める。
        *   特に `PlayerManager.cs` の修正は、`PlayerMover` と `PlayerAttacker` のインスタンス化と初期化ロジックに影響するため、慎重に行う。

6.  **ToDo 6: コード修正の動作確認** (保留中)
    *   実装された機能が意図通りに動作するかをテスト。
    *   **考慮事項**:
        *   計画されたテストシナリオを網羅的に実施する。
        *   デバッグツールやログを活用し、内部状態の変化を追跡することで、問題の原因特定を迅速に行う。
        *   Unityエディタのインスペクタで、`PlayerMover.CurrentVelocity` の値や `PlayerAttacker.IsAttacking` の状態を確認できるとデバッグが容易になる。

## 3. 全体的な考慮事項

*   **段階的な実装**: 一度に全ての変更を適用するのではなく、各ToDoのステップを完了するごとに動作を確認することを推奨します。
*   **既存コードの尊重**: 既存のアーキテクチャやデザインパターンを理解し、それに沿った変更を心がけます。
*   **パフォーマンス**: 移動や攻撃は頻繁に実行される処理であるため、パフォーマンスに影響を与えないよう効率的な実装を心がけます。
*   **拡張性**: 将来的に他のアクション（回避など）が追加された場合でも、容易に拡張できるような設計を意識します。
